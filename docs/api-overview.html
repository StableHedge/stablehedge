Schema Reading Points

도메인 해석

  • Fund는 장기 존속하는 투자상품이다.
  • Investment가 펀드와 투자자를 연결하고 지분율을 들고 있다.
  • Distribution은 분기별 또는 이벤트성 배당 batch다.
  • DistributionItem은 투자자 1명분 개별 지급 건이다.

운영 해석

  • Distribution.status는 전체 batch 상태다.
  • DistributionItem.paymentStatus는 개별 지급 상태다.
  • XrplTransaction은 submit 이후의 온체인 감사 로그다.
  • 즉, 하나의 배당 batch 안에 여러 investor payment가 들어간다.

이 슬라이드 다음 설명 순서

Fund → Distribution 생성 (DRAFT) → Calculate로 DistributionItem 생성 (READY) → Submit 시 investor별 XRPL Payment 제출 → 결과를 XrplTransaction + DistributionItem.paymentStatus에 반영

Distribution.status

DRAFT
  │  POST /api/distributions/:id/calculate
  ▼
READY
  │  POST /api/distributions/:id/submit
  ▼

  ▼

  │  ledger validation
  ├──▶ VALIDATED
  └──▶ FAILED

Calculate Distribution — "오프체인" 계산

POST/api/distributions/:id/calculate

Dashboard의 Calculate Distribution 버튼. XRPL을 건드리지 않는다. DB 안에서 지분율로 USD를 쪼갠다.

핵심 공식

amountUsd = totalDistributableUsd × ownershipBp / 10000
amountKrw = amountUsd × fxRate    // fx 있을 때만

basis-point: 5000 → 50.00%

가드

  • status가 SUBMITTED/SETTLED면 거부 (재계산 금지)
  • 같은 distributionId의 기존 items 전부 삭제 후 재생성 (atomic)

흐름 (distribution-engine.ts)

distribution + investments 로드
   │
   ▼
computeDistributionAmounts(total, [{investorId, ownershipBp}], fx)
   │  (distribution-math.ts, 순수 함수)
   ▼
prisma.$transaction:
   ├─ distributionItem.deleteMany
   ├─ distributionItem.createMany (status=PREPARED)
   └─ distribution.update { status: READY, calculatedAt }

Response

{ "id":"...", "status":"READY", "itemCount":3, "calculatedAt":"..." }

어디서 XRPL이 필요한가

이 프로젝트의 본질은 오프체인 금융 운영 플랫폼이다. XRPL은 모든 단계에 필요한 것이 아니라 정산 실행과 증빙 단계에서만 필요하다.

XRPL이 굳이 필요 없는 영역

  • 환노출 분석, 환율 시나리오 계산
  • 지분율 기준 배당 금액 계산
  • 투자자 리포트 생성
  • 헤지 계약 등록 및 손익 계산
  • 운영 대시보드와 조회 API

이 영역은 Fastify + Prisma + PostgreSQL만으로 충분하다.

XRPL이 필요한 핵심 순간

  • 투자자별 토큰 지급을 실제로 실행할 때
  • txHash, ledger status, explorer link를 남길 때
  • 온체인 지급 증빙을 Investor Statement에 연결할 때
  • Ledger Watcher로 제출 결과를 재동기화할 때

즉, POST /api/distributions/:id/submit이 XRPL settlement rail에 진입하는 유일한 관문이다.

한 줄 구조

Create Distribution → Calculate Distribution → Submit Distribution DB 기록 / 환율 적용 DB 계산 / READY 전환 XRPL Payment 제출 / txHash / validation

Submit — 시퀀스 다이어그램

Route Service xrpl-adapter xrpl.js / WSS DB ───── ─────── ──────────── ───────────── ── POST /:id/submit ─▶ submitDistribution() │ load + status guard │ decryptSeed(treasury) ─────────────────────────────────────────▶ wallet-vault │ status=SUBMITTED, submittedAt ─────────────────────────────────▶ Distribution │ ▼ submitDistributionPayments() ─▶ withClient(WSS) client.connect() for each payment: │ build Payment tx │ │ client.autofill(tx) │ wallet.sign(prepared) ──▶ txHash 결정 (오프라인) onSubmitted({txHash}) ◀─────────┤ ├─ item.paymentStatus=SUBMITTED, txHash ─────────────────────▶ DistributionItem └─ XrplTransaction.upsert(SUBMITTED) ────────────────────────▶ XrplTransaction │ │ client.submitAndWait(blob) │ ⋯ ledger validation ⋯ │ ◀── meta.TransactionResult result: VALIDATED | FAILED ◀────┤ per-item update ────────────────────────────────────────────────▶ DistributionItem XrplTransaction │ (loop end) │ ▼ all VALIDATED → status=SETTLED any FAILED → status=FAILED ─────────────────────────────────────▶ Distribution

🟢 핵심 트릭: sign 직후 txHash가 결정되므로 검증 응답 도착 전에 미리 DB에 SUBMITTED+txHash를 기록한다(onSubmitted 콜백). 서버가 죽어도 ledger-watcher가 이어받아 동기화 → 멱등성.

XRPL Settlement Monitor

GET/api/settlement-monitor/funds/:fundId

펀드 단위 대시보드. ?page=1&pageSize=5로 최근 트랜잭션 페이지네이션.

잔액 산출 로직

// "투자자 잔액" = 검증된 분배의 합산
settledItems = DistributionItem
  where fund=fundId AND paymentStatus IN (VALIDATED,REFLECTED)
balanceByInvestor = group(settledItems).sum(amountUsd)

investorBalancesTotal = Σ balanceByInvestor
treasuryBalance       = totalIssued − investorBalancesTotal

// "이번 송금 전" 추정치
beforePayout = treasuryBalance + latestTxAmount
change       = treasuryBalance − beforePayout

잔액 전용 ledger 추가 전까지의 라이트 프록시. mainnet 가면 account_lines로 교체 예정.

사이드 채널 — Ledger Watcher

HTTP API는 아니지만 모든 API의 정확성을 떠받치는 백그라운드 컴포넌트.

역할

  • 서버 부팅 시 startLedgerWatcher() 1회
  • 모든 펀드의 treasuryAddress를 WSS 구독
  • 새 트랜잭션 수신 → XrplTransaction upsert
  • validated 이벤트 → 매칭되는 DistributionItem을 VALIDATED로 승격

왜 필요한가

  • 멱등성: /submit이 응답 받기 전에 죽어도 ledger가 진실. 재기동 시 watcher가 따라잡음
  • 외부 입금: 추후 redemption/입금이 있어도 자동 기록
  • 감사: rawResult에 ledger 응답 통째로 보관
// xrpl.js 사용
client.request({ command: "subscribe", accounts: [...treasuries] })
client.on("transaction", (event) =>
  prisma.xrplTransaction.upsert({ where: { txHash }, ... })
)

현재 미구현 기능 — Admin / Onboarding 영역

데모 1주 스코프에서 잘라낸 영역. 시드 스크립트나 하드코딩으로 대체됨. 본질적으로 B2B 운용사·관리자 책임의 흐름이라 사용자 노출 API보다는 Admin/내부 시스템에 가깝다.

투자자 Onboarding (Admin)

현재: prisma/seed.ts에 투자자 3명 + ownershipBp 5000/3000/2000 하드코딩

  • POST /api/investors — 투자자 등록
  • POST /api/funds/:fundId/investments — 펀드 가입 + 지분(bp) 부여
  • PATCH /api/funds/:fundId/investments/:id — 지분 변경
  • POST .../investments/:id/open-trustline — Trustline 자동 개설

실서비스 시 외부 KYC/AML 시스템과 묶여서 백엔드는 결과만 받는 형태가 자연스러움

운영 (Operator) 도구

  • POST /api/admin/fx-rate — 환율 수동 입력 (현재 1468 demo-fixed)
  • FX 자동 갱신 — cron / 외부 API / XLS-47 온체인 오라클
  • bp 합계 검증 정책 (현재 미검증, 11000 가능)
